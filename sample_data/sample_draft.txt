RubricLens: A Rubric-Aligned Draft Self-Evaluation Tool

Introduction and Project Aims

This project, RubricLens, was developed as part of the CM2020 Agile Software Projects module. The primary aim was to create a tool that helps students evaluate their coursework drafts against a marking rubric, providing formative feedback and identifying gaps in coverage before final submission.

The core objectives were:
1. Allow students to input or import marking rubrics with criteria and performance descriptors.
2. Enable draft text submission via paste or document upload.
3. Use text retrieval techniques to automatically identify evidence in the draft for each rubric criterion.
4. Generate an explainable coverage report with actionable improvement suggestions.
5. Export reports in multiple formats for offline review.

Technical Approach and Architecture

The application follows a client-server architecture with a Python FastAPI backend and a vanilla HTML/CSS/JavaScript frontend. This approach was chosen to keep dependencies minimal while demonstrating core web development skills.

The backend is structured into distinct modules:
- database.py handles SQLite operations with a well-defined schema
- text_processing.py manages text normalisation and sentence-aware chunking
- retrieval.py implements TF-IDF evidence matching using scikit-learn
- report_generator.py applies deterministic rules for status classification
- export.py generates Markdown and PDF reports

The database uses SQLite for zero-configuration local-first operation, which aligns with the privacy requirement of not sending student work to external services. Foreign key constraints with cascade deletes ensure referential integrity.

Iterative Design Process

The development followed an iterative approach across multiple sprints. In the first sprint, the focus was on establishing the project structure, database schema, and core text processing pipeline. User feedback from an initial prototype revealed that the chunking algorithm needed refinement to handle paragraph boundaries more gracefully.

In the second sprint, the TF-IDF retrieval engine was implemented and tuned. Initial testing showed that the default relevance threshold was too high, causing many valid evidence matches to be filtered out. The threshold was lowered from 0.10 to 0.05 based on testing with sample drafts.

The third sprint focused on the report generator and export functionality. Feedback from peer review sessions led to improvements in the rationale text, making it more specific by referencing rubric descriptor language rather than generic statements.

The fourth sprint addressed frontend implementation and UX polish. A formative evaluation with three student stakeholders identified several usability issues: the rubric editor needed inline descriptor editing, the analysis loading state needed a spinner, and the evidence drill-down needed to show rubric descriptors alongside evidence excerpts.

Testing Strategy

The testing regime uses pytest as the test framework, chosen for its simplicity and powerful fixture system. Tests are organised into separate files by module:
- test_database.py: 21 tests covering all CRUD operations and cascade deletes
- test_text_processing.py: 25 tests for normalisation, chunking, and docx extraction
- test_retrieval.py: 12 tests for TF-IDF retrieval accuracy and heuristic filters
- test_report_generator.py: 25 tests for status classification and rationale generation
- test_export.py: 11 tests for Markdown and PDF export
- test_api.py: 19 tests for API endpoint responses and error handling

The testing approach is justified by the need for regression testing during iterative development. Unit tests ensure individual components work correctly, while API tests verify end-to-end behaviour through the HTTP layer. pytest-cov tracks code coverage to identify untested code paths.

Error handling is implemented throughout the application with try-except blocks in the backend and user-friendly error messages in the frontend. The API returns appropriate HTTP status codes: 400 for bad input, 404 for not found resources, and 500 for server errors.

User Evaluation

A formative evaluation was conducted with five representative stakeholders (fellow students taking the CM2020 module). Each participant was asked to:
1. Load the demo rubric
2. Paste a sample draft
3. Run the analysis
4. Review the coverage report
5. Export a PDF report

Feedback was collected through a structured questionnaire covering usability, clarity of the coverage report, and usefulness of the next-action suggestions. Key findings included:
- All participants could complete the primary workflow without guidance
- The colour-coded status badges were immediately understood
- Two participants suggested adding a progress bar during analysis
- Three participants found the evidence drill-down helpful for understanding their gaps

A summative evaluation measured task completion time and error rates. The average time to complete the full workflow was 3.2 minutes, and all participants completed the task without errors. The System Usability Scale score was 78, indicating good usability.

Technical Challenge and Novelty

The technical challenge of this project lies in implementing a functional text retrieval system that can meaningfully match rubric criteria to draft content without relying on external AI APIs. The TF-IDF approach provides deterministic, explainable results that can be inspected and understood by users.

Novel aspects include the heuristic filtering system that detects and downweights generic chunks that match too many criteria, the deterministic status classification rules that ensure reproducible results, and the integrated approach of combining text retrieval with rubric-specific report generation to provide actionable feedback.

The system successfully handles drafts of up to 5000 words within the 10-second performance target, processing text through normalisation, chunking, TF-IDF vectorisation, and report generation in a single pipeline.

Code Quality and Structure

The codebase follows a sensible structure with clear separation of concerns. Each backend module has a single responsibility: database operations, text processing, retrieval, report generation, and export. The frontend uses a single-page application pattern with vanilla JavaScript for DOM manipulation and API calls.

Code style is consistent throughout, with docstrings on all public functions, meaningful variable names, and consistent formatting. The project avoids unnecessary dependencies, using only well-established libraries for specific purposes.
